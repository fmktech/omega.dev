Product Requirements Document · v0.2

Omega.dev

A self-improving software-engineering agent that turns costly successful problem-solving into safely tested, reusable harness knowledge—connected to any compatible model provider or fully local compute.

Draft for review Provider or local 10 July 2026
Architecture status: this v0.2 PRD is historical product context. Accepted records in docs/adrs/ are the implementation source of truth where they conflict with this document, including automatic project-scoped promotion, mutable generational evaluation, safe-boundary harness activation, filesystem-native storage, and skill-like project knowledge.

Product thesis

Omega is not an agent that rewrites its live runtime. It is a coding agent with a separate, durable evolution loop: task traces expose repeatable breakthroughs; bounded subagents distill them into candidate improvements; independent evaluation decides whether a project, domain, or global harness bundle should change.

Operating principle

Pay for a difficult solution once; retain only the transferable method.

A successful patch is evidence. A verified procedure that helps earlier on related tasks is a reusable asset.

Problem statement

Software-engineering agents repeatedly rediscover techniques that they have already expensively learned. A long, exploratory trajectory may eventually solve an issue, but the useful turning point is trapped inside an unstructured trace. The next task—possibly in the same repository—pays the same cost again.

Existing agent harnesses also tend to be either fixed and generic, or dangerously self-modifying. Fixed harnesses cannot learn from repeated project experience. Unbounded self-modification risks corrupting permissions, weakening evaluation, leaking project information, or optimizing for an artificial benchmark instead of real engineering quality.

Omega must serve many projects without treating them as one shared context. It needs to preserve useful project knowledge, selectively generalize proven methods, and remain fully operable on a local machine without dependence on Vercel-managed infrastructure.

Solution

Omega is a CLI-driven SWE agent with two first-class model modes: any compatible model provider or fully local compute. It runs tasks in isolated sandboxes, records durable traces, and uses a controlled evolve harness workflow to turn recurring failures and costly breakthroughs into versioned, scope-aware harness bundles.

Goals

Primary goals

  • Offer provider freedom: connect any compatible model provider or run fully local compute.
  • Capture high-value successful trajectories and crystallize transferable methods.
  • In v0, learn only at project scope; earn any later domain or global capability through separately designed cross-project evaluation.
  • Make every evolution proposal reproducible, inspectable, reversible, and attributable.
  • Use one named Promotion Eval protocol to compare incumbent and candidate harnesses against the same evidence and decision rule.
  • Use AI SDK v7 as a provider-neutral model router while agent code refers only to stable logical model roles.
  • Gate every side-effecting action through a tunable execution-policy decision that can allow, deny, or escalate it.
  • Offer a first-class terminal experience rather than exposing framework-specific commands.

Non-goals for v0

  • Unrestricted live self-modification of runtime code, tool permissions, or sandbox policy.
  • Domain/global promotion, automatic promotion, or cross-project learning in v0.
  • Mandatory use of Vercel, Vercel AI Gateway, or arbitrary third-party model-provider adapters.
  • Replacing a project’s CI, code review process, or human ownership model.
  • Training or changing base-model weights.
  • Claiming that a passing test suite alone establishes maintainable software quality.

Users and user stories

Primary users

Local developer

Runs Omega in one or more repositories and wants faster, more reliable assistance without surrendering project data or control.

Agent operator

Configures models, safety boundaries, budgets, sandbox backends, retention, and promotion policy.

Harness researcher

Studies which workflows, skills, and model routes reliably improve engineering outcomes across a portfolio of projects.

Detailed user stories

  1. As a local developer, I want to ask Omega to solve a repository task from the terminal, so that I can use it without a hosted control plane.
  2. As a local developer, I want Omega to run commands and tests in an isolated sandbox, so that a task cannot silently alter my host environment.
  3. As a local developer, I want Omega to retain a compact project profile, so that it remembers validated commands, conventions, and pitfalls when I return.
  4. As a local developer, I want project knowledge tied to a repository fingerprint and freshness marker, so that stale facts are revalidated rather than blindly trusted.
  5. As a local developer, I want to inspect the trace behind a proposed lesson, so that I can understand why Omega believes it is reusable.
  6. As a local developer, I want to trigger harness evolution from a completed run, so that hard-won breakthroughs can compound into future performance.
  7. As a local developer, I want evolution to run asynchronously, so that normal task work is not blocked by exploratory subagents.
  8. As a local developer, I want to see the pinned project harness version and its parent evidence, so that behavior is explainable and reproducible.
  9. As a local developer, I want to pin or roll back a harness version, so that an unwanted behavioral change is reversible immediately.
  10. As an operator, I want to choose any compatible model provider or local compute, so that I retain control over capability, cost, privacy, and model availability.
  11. As an operator, I want each mode to make its network, cost, capability, and data-handling policy explicit, so that the choice stays under my control.
  12. As an operator, I want model calls to use logical roles instead of raw model names, so that routing can change without rewriting agent behavior.
  13. As an operator, I want every run to record the resolved model route and generation settings, so that outcome comparisons are meaningful.
  14. As an operator, I want a model allowlist, cost budget, and capability policy, so that evolution cannot silently expand provider access or spend.
  15. As an operator, I want the Promotion Eval route pinned independently of candidate routing, so that a candidate cannot optimize by manipulating its own judge.
  16. As an operator, I want to tune an execution-policy profile from manual to guarded to autonomous, so that routine low-risk actions can proceed while meaningful risk is still escalated.
  17. As a project owner, I want every proposed side effect to receive an explainable allow, deny, or approval-required verdict, so that I can audit why an action did or did not run.
  18. As a project owner, I want hard-deny rules and mandatory approval rules to override the policy model, so that a cheap fast model never grants authority beyond my explicit policy.
  19. As a harness researcher, I want Omega to distinguish failure clusters from one-off task difficulty, so that mutations address mechanisms rather than symptoms.
  20. As a harness researcher, I want Omega to identify the turning point in a successful grind, so that it captures the method rather than copying a task-specific answer.
  21. As a harness researcher, I want proposed changes to be narrow and diverse, so that experimentation remains interpretable and avoids duplicate variants.
  22. As a harness researcher, I want candidates to replay relevant pre-breakthrough traces, so that the proposed lesson proves it would have helped earlier.
  23. As a harness researcher, I want candidates tested through a sealed project Promotion Eval containing both target and regression tasks, so that fixing one known weakness does not regress general behavior.
  24. As a harness researcher, I want one Promotion Eval to run incumbent and candidate harnesses on the same frozen task conditions, so that a promotion decision measures a real behavioral delta rather than task luck.
  25. As a harness researcher, I want each Promotion Eval to require a configured minimum effect threshold and record cost and latency, so that small or expensive apparent wins do not compound into the active harness.
  26. As a harness researcher, I want an abstract lesson to begin project-scoped, so that one project’s quirks do not become global behavior.
  27. As an operator, I want cross-project promotion disabled in v0, so that Omega first proves safe project-scoped compounding before it attempts broader transfer.
  28. As a project owner, I want raw source, secrets, and issue content to remain project-local, so that global learning cannot become a data-leak path.
  29. As a project owner, I want global lessons to contain only redacted, abstracted procedures, so that shared knowledge preserves confidentiality boundaries.
  30. As a project owner, I want privileged actions to remain approval-gated, so that agent learning cannot bypass human authority.
  31. As a project owner, I want Omega to preserve rejected candidates and negative results, so that the system does not repeat failed experiments.
  32. As a CLI user, I want concise run, evolution, memory, experiment, and promotion commands, so that routine operation does not require a web application.
  33. As a CLI user, I want streaming status and a readable final decision summary, so that durable background work remains visible from the terminal.

Implementation decisions

  • Provider freedom or local compute Required Omega supports any compatible model provider through direct or OpenAI-compatible adapters, and also supports fully local compute. The operator owns provider selection, credentials, network policy, and model availability; Omega never introduces a proprietary model gateway or an implicit Vercel dependency.
  • Eve-inspired agent organization Required Omega adopts the separation of instructions, tools, skills, hooks, subagents, sandbox definition, evaluations, schedules, and channels. This is an organization specification, not a runtime dependency on Eve.
  • AI SDK v7 model router Required All model access passes through an explicit AI SDK provider registry and model-router policy. AI SDK is the internal abstraction that lets Omega use configured direct providers and local endpoints without Vercel AI Gateway as an implicit default.
  • Execution Policy Agent Required Before a side-effecting tool call runs, a deterministic gate evaluates hard policy and sends a structured action intent to a cheap, fast policy-model role. Its verdict is allow, deny, or escalate, with confidence, reasons, and any constraints. Hard-deny rules, mandatory approvals, and low-confidence decisions always override an allow verdict.
  • Tunable autonomy profiles Required Operators choose a bounded manual, guarded, or autonomous profile at global and project scope. A child scope may narrow autonomy but may never exceed the protected maximum set by its parent. Irreversible, sensitive, or externally consequential action classes remain non-auto-approvable.
  • Logical model roles Required Task agents, evolution proposers, triage workers, embeddings, Promotion Eval, policy-agent, and promotion-agent roles ask for named capabilities and budgets. The router resolves each role to a pinned configured provider/model route and records the resolution in the trace.
  • Trusted control plane Protected Sandbox policy, execution-policy hard rules and maximum autonomy, permissions, provider credentials, Promotion Eval, policy-agent and promotion-agent routes, retention policy, and promotion rules are not editable by the agent evolution loop.
  • Mutable harness bundles Required Evolution produces immutable, content-addressed bundles containing only approved mutable surfaces: skill/playbook content, contextual retrieval rules, bounded workflow strategy, and task/mutator role-selection policy. Bundles cannot alter Promotion Eval, policy-agent, or promotion-agent routing.
  • Remote-style local registry, not local Git as source of truth Required The authoritative history is an append-only registry of bundle manifests, parent relationships, trace artifacts, evaluation records, and promotion pointers. Ephemeral workspaces may exist for task execution but do not version the harness.
  • Scope hierarchy Required V0 composes protected core policy, one project bundle, and task-local scratch state. Domain and global bundle layers are architecture placeholders only: they remain disabled until a separate cross-project evaluation and privacy design is approved. No scope can override protected controls.
  • Project profile as bounded memory Required Each project has an access-controlled, compact profile containing repository identity, last validated state, known-good commands, confirmed conventions, verified pitfalls, linked lessons, confidence, and retention metadata. Every entry records the source run and verification that justified it; new entries are unverified until executed successfully, decay over time, and are policy-gated writes.
  • Crystallization before broad mutation Required A costly successful trajectory first becomes a candidate procedural lesson. It becomes a project harness mutation only after the Promotion Eval demonstrates that it improves a reusable method rather than memorizing an answer.
  • Promotion Eval Required Promotion Eval is Omega’s single promotion-evidence protocol, not a separate feature. It runs the incumbent and candidate harnesses against the same frozen project task conditions and pinned route class, compares success, verification quality, cost, and latency, then applies configured replicate and minimum-effect rules. A sealed project eval set is the initial mode; prospective evaluation on future real tasks is the second mode when replay is not available.
  • Route-bound evidence Required A harness bundle’s evidence is bound to its model-route class. Changing the task or mutator route marks the bundle unverified for that route until it passes the Promotion Eval again. Local mode must meet an explicit evaluation/diagnostician capability floor before evolution is enabled.
  • Bounded evolution workflow Required The main agent invokes an asynchronous evolution tool with evidence, a goal, allowed surfaces, scope constraints, and a budget. The tool coordinates diagnostician, mutator, promotion-evaluation, and auditor roles in isolated executions.
  • Independent promotion policy Protected A candidate must meet its Promotion Eval’s paired, minimum-effect, cost, and no-regression criteria. V0 promotion is always human-approved and project-scoped; it is a policy decision, not an agent action.
  • Privacy-preserving generalization Required Raw project artifacts remain scoped. Any proposed cross-project lesson must be abstracted and redacted before it enters a shared candidate bundle.

Promotion Eval

Promotion Eval is the evidence system behind every harness promotion. It compares a candidate against the active incumbent under matched conditions; it is not a second product loop or a vague “held-out check.”

Frozen project task snapshot + verifier + pinned route class + fixed resource policy │ ┌─────────────┴─────────────┐ ▼ ▼ incumbent harness run candidate harness run │ │ └─────────────┬─────────────┘ ▼ paired scorecard: verified outcome · cost · latency · policy conformance │ ▼ configured replicates + minimum effect threshold + no-regression rule │ ▼ human-approved project promotion / rejection / canary

Sealed project mode

Before a promotion, Omega evaluates on a compact, content-hash-sealed project task set. Task snapshots, verifier configuration, and route class are pinned. Diagnostician and mutator roles receive results, not hidden task contents.

Prospective mode

When historical replay is unavailable, Omega tests a manually approved candidate on the next eligible real tasks alongside an incumbent baseline. This is a second mode of the same Promotion Eval, not a weaker untracked learning path.

Decision rule

Each scorecard records matched runs, verified task outcome, cost, latency, route class, and policy behavior. A candidate must clear configured replicate, minimum-effect, and no-regression thresholds before a human may promote it.

Scope of truth

Evidence proves only the tested project bundle under the tested route class. A route change, composition change, or future cross-project scope marks the lesson unverified until a relevant Promotion Eval succeeds.

V0 truthfulness rule: if Omega cannot reproduce an earlier task from a frozen task-start snapshot, it does not claim that a lesson would have changed that historical trajectory. It uses sealed or prospective Promotion Eval instead.

Execution Policy Agent

Omega treats approval as a runtime decision, not an afterthought. The task agent proposes an action, but the execution-policy gate bases enforcement on the actual tool invocation and observed sandbox capabilities—not on the agent’s self-description.

Task agent proposes an action │ ▼ Action adapter derives actual command, paths, mounts, egress, and side effects │ ▼ Deterministic policy check ── hard deny / mandatory approval? ──► deny or escalate │ ▼ Fast Policy Agent (cheap, pinned model role) │ ▼ allow · deny · escalate + confidence + reasons + constraints │ ▼ Protected decision merger ──► executor or human approval request

Action intent

Every proposed side effect is normalized from the actual tool operation: command, target paths and hosts, mounts, egress, expected diff, project scope, and existing approval. The agent’s stated purpose is optional context, never trusted evidence.

Fast policy role

The policy agent uses a small, low-latency, structured-output model role. It classifies contextual risk from observed action facts that static rules cannot fully capture, but it is not itself the source of authority.

Decision semantics

Allow permits only an action class already eligible for automatic execution. Deny prevents the action. Escalate pauses work and asks the configured human or higher-authority policy for a decision.

Autonomy profiles

Manual escalates all side effects. Guarded auto-allows reversible, project-local low-risk work. Autonomous expands eligible classes within hard caps; it never bypasses protected denials or mandatory approval classes.

Safety invariant: model confidence and agent-reported intent are not permission. An allow result is effective only when observed sandbox behavior and deterministic policy also permit the action; uncertainty defaults to escalation rather than silent execution.

Evolution and crystallization workflow

Evolution is not invoked for every task. It begins after an explicit request, a verified repeated failure pattern, or a high-cost successful run whose trace contains a plausible turning point.

Collect evidence

Store the pinned harness version, resolved model route, task context, tool outcomes, verification results, user approvals, and final artifact. Mark both failures and expensive successes as candidates for analysis.

Mine mechanism, not symptom

A diagnostician separates terminal outcome from causal behavior. “Timeout” is not enough; the system identifies the decision, missing context, workflow gap, or tool-use pattern that caused it.

Extract the turning point

For a successful grind, identify the observation or procedure that transformed the search. Remove repository-specific facts and state the smallest reusable method.

Propose bounded candidates

Mutator subagents create diverse candidate bundle changes within declared editable surfaces. They may not edit protected runtime or Promotion Eval components.

Replay and transfer-test

Run the Promotion Eval. If a frozen task-start snapshot and sufficient checkpoint exist, compare incumbent and candidate on it; otherwise use the sealed project set or prospective mode. The system never substitutes an after-the-fact narrative for measured replay evidence.

Audit and promote

Accept only project-scoped candidates that clear the Promotion Eval’s effect, regression, cost, privacy, route, and policy rules, then obtain human approval. Record rejected candidates and negative results so failed avenues are not re-run blindly.

Anti-overfitting rule: a task solution does not become an active project lesson because it passed once. It must clear the Promotion Eval; domain and global promotion are disabled in v0.

CLI experience

The CLI is Omega’s primary product surface. It is a thin local client to the runtime and registry, not a replacement for the agent’s sandbox or a wrapper around Vercel tooling.

Core task operations

  • Start a task in the current project.
  • Stream task and tool progress in a concise terminal view.
  • Inspect a trace, resolved harness bundle, model route, and verification result.
  • Resume or cancel a durable local task where policy permits.

Evolution operations

  • Request evolution from a run, failure cluster, or explicit goal.
  • List, inspect, compare, and cancel experiments.
  • Run or inspect a Promotion Eval showing incumbent/candidate results, route class, effect threshold, and cost.
  • Promote, pin, or roll back a scoped bundle subject to policy.

Memory and policy operations

  • Inspect, refresh, or forget a project profile.
  • View data-retention scope and linked lessons.
  • Inspect effective model routing and configured budgets.
  • Export a local, redacted experiment report.

Operator operations

  • Configure local sandbox backend, approved providers, and model allowlists.
  • Set spending, time, concurrency, and network policies.
  • Manage Promotion Eval suites and promotion requirements.
  • Back up or migrate the local registry without relying on a hosted platform.

Testing decisions

Tests should assert externally observable behavior and durable invariants, not incidental internal implementation. Every test fixture must pin its harness bundle, Promotion Eval configuration, model route or deterministic substitute, and sandbox policy.

Unit and contract tests

  • Model-router capability, allowlist, budget, fallback, and attribution behavior.
  • Execution-policy decision precedence: hard deny and approval requirements always override policy-model allow results.
  • Execution-policy adapters derive decisions from observed command, mount, path, and egress facts rather than agent-reported intent.
  • Scope resolution, precedence, freshness, provenance, expiry, and retention behavior for project memory.
  • Bundle lineage, content addressing, promotion atomicity, and rollback correctness.
  • Policy enforcement that rejects edits to protected surfaces, including Promotion Eval, policy-agent, and promotion-agent routes.
  • CLI command contracts and machine-readable status output.

Integration tests

  • Local daemon, database, artifact store, CLI, and Docker/Podman sandbox in one repeatable environment.
  • Action intents are logged and an escalation reliably pauses execution until a configured approval arrives.
  • Durable evolution jobs survive interruption and preserve inspectable partial evidence.
  • Multiple subagents remain isolated and cannot share unapproved project artifacts.
  • Local-provider execution works without a network gateway.
  • Project profile storage and deletion follow configured data scope.
  • A route change invalidates prior candidate evidence until the relevant Promotion Eval re-runs.

Evaluation tests

  • Sealed project task snapshots measure incumbent/candidate verified success, verification quality, cost, and latency under the same route class.
  • Promotion Eval uses configured matched replicates, a minimum effect threshold, a no-regression rule, and a fixed evaluation-compute budget.
  • Promotion Eval runs the exact effective bundle composition; any later composition change invalidates its scorecard until re-evaluated.
  • Prospective mode tracks a manually approved candidate on future eligible project tasks and can quarantine it on regression.
  • Historical replay is tested only where a frozen task-start snapshot and sufficient checkpoint exist; otherwise the scorecard states that replay was unavailable.
  • Adversarial suite probes Promotion Eval gaming, test overfitting, privilege escalation, and data leakage.
  • Adversarial suite includes repository-content prompt injection against the Execution Policy Gate.
  • Policy calibration suite measures false allows, false denials, escalation rate, latency, and policy-model cost by action class.

Acceptance criteria

  • A promoted candidate has a complete, inspectable lineage and scorecard.
  • Its Promotion Eval clears the configured paired-run, minimum-effect, and no-regression requirements.
  • Its bound model route and resource cost remain inside policy.
  • It contains no protected-surface change or prohibited project data.
  • It is project-scoped and manually approved in v0.
  • Rollback restores the previous effective behavior deterministically.

Safety, privacy, and governance

Permission boundary

Evolution agents may read approved traces and modify only declared candidate surfaces in isolated environments. They cannot broaden network access, modify host policy, install credentials, alter Promotion Eval, or activate a candidate directly.

Execution boundary

The Execution Policy Agent classifies every side effect, but deterministic rules and configured human gates retain final authority. Policy-model failures, timeouts, and uncertainty resolve to denial or escalation according to the active profile.

Privacy boundary

Project profiles and traces are local by default. Promotion beyond a project requires a redaction pass, provenance, and evidence that the result is procedural rather than proprietary.

Reward boundary

Promotion Eval combines matched verification outcomes with regression, cost, latency, policy, route, and audit signals. Sealed task contents are unavailable to the roles proposing candidates; the system must never treat an agent-authored score change as sufficient proof of improvement.

Human boundary

Configured approval gates remain available for repository writes, external side effects, domain/global promotion, and any risk category the operator chooses.

Phased rollout

Phase 0 — Contract and local baseline

Define the agent directory specification, trusted/mutable boundary, local registry schema, CLI vocabulary, protected model roles, frozen task-start snapshot format, sealed project eval-set format, Promotion Eval metrics, and a reproducible baseline task harness.

Phase 1 — Useful local SWE agent

Deliver task execution, sandbox integration, trace capture, provider-agnostic/local model routing, the Execution Policy Agent, provenance-aware project profiles, Promotion Eval scorecards, and the core CLI.

Phase 2 — Inspectable evolution

Add asynchronous evolution jobs, trace distillation, narrow project-scoped skill candidates, sealed and prospective Promotion Eval modes, experiment inspection, manual promotion/rollback, and post-promotion canary/quarantine.

Phase 3 — Evaluate the case for transfer

Only if project-scoped lessons show positive realized savings, design a separate cross-project evaluation, privacy, redaction, and human-approval system before enabling domain bundles or scope escalation.

Phase 4 — Bounded workflow and router evolution

Allow candidates to improve approved workflow strategy and task/mutator model-role selection within fixed resource and Promotion Eval constraints. Maintain a diverse candidate population only if evaluation quality supports it.

Out of scope

  • General autonomous software deployment without configured approval and repository policy.
  • Editing model weights, fine-tuning pipelines, or claiming recursive improvement of base-model intelligence.
  • Cross-tenant or cross-user global learning without explicit policy, redaction, and consent mechanisms.
  • Domain/global promotion, automatic promotion, or arbitrary mid-trajectory replay in v0.
  • Distributed cloud orchestration as a requirement for v0.
  • Bespoke first-party integration work for a provider that lacks an AI SDK-compatible or OpenAI-compatible adapter; provider adapters remain a pluggable extension point.
  • Replacing expert review of maintainability, product intent, security architecture, or organizational ownership.

Further notes and open decisions

Local model baseline

Select a first supported local model/runtime combination and define the minimum tool-calling, context, and structured-output capabilities required for Omega’s task, diagnostician, and Promotion Eval roles.

Sandbox default

Select Docker or Podman as the first backend and define the default network, filesystem mount, image provenance, and resource limits.

Evaluation corpus

Define a private, representative, content-hash-sealed starting corpus of repository tasks and regression scenarios. A trustworthy eval set is a prerequisite for meaningful evolution.

Promotion Eval calibration

Set the initial matched-replicate count, minimum effect threshold, no-regression rule, false-promotion tolerance, and evaluation-compute budget. These are promotion-policy inputs, not values the evolving harness may change.

Promotion authority

V0 is fixed: only project-scoped, human-approved promotion. Decide later whether a measured canary can reduce review burden without permitting autonomous or cross-project promotion.

Execution-policy baseline

Define the initial hard-deny classes, mandatory approval classes, autonomy profiles, and the quality/cost target for the fast policy-model role. A conservative default is guarded mode with automatic execution only for reversible project-local actions.

Retention and portability

Define default retention duration, encryption-at-rest expectations, selective deletion semantics, and a portable backup/export format for local users.

Definition of domain

Define whether domains are manually configured, inferred from project metadata, or both. Initial promotion should prefer explicit domain membership over inference alone.

Omega.dev PRD v0.2. This document captures the current product direction: an Eve-inspired, AI SDK v7-routed SWE agent that works with any compatible model provider or fully local compute, whose project-scoped evolution is evidence-driven by Promotion Eval and bounded by a trusted control plane.
